# <昵称>命盘简洁版

source_id: `SRC-PROJECT-LONGFORM-TEMPLATE`

## 用途

简洁版是完整交付里的读者快速阅读版本，不替代丰富版。它面向普通读者，要求能直接发送、容易读完、没有过程文字，但仍然保留关键命盘锚点和现实建议。

执行边界：不能因为本次资料只含八字、只含流盘截图、缺紫微或缺西占，就把丰富版改成只交付简洁版。资料缺模块时，丰富版按可用 facts 写成单体系/可用体系长文；简洁版只能作为附加交付或用户明确要求的短版。

默认长度：5500-9000 字符。低于 5000 通常不够丰富；高于 10000 通常应改回丰富版或拆成完整长文。

## 写作原则

- 全文默认第二人称“你”，除非用户明确要求第三方视角。
- 不使用“小红书”“情绪价值”“真实感增强”等内部讨论词作为标题、文件名或正文标签。
- 不写脚本路径、JSON 路径、执行命令、截图来源、坐标来源、验收过程或交付方说明。
- 技术事实只做短锚点，不堆证据。推荐只写 1 段“专业锚点”，包含四柱/大运/紫微/西占里最关键的稳定项。
- 语言要有代入感，但不按性别套话。女性、男性和中性读者都按命盘结构调整表达。
- 每段至少新增一个判断、场景、风险、动作或校准问题，避免把同一句话换词重复。
- “边界”“表达”“需求说出来”不能作为默认人设。只有当八字十神、紫微宫位/四化、西占落座/相位或读者校准同向时，才写成强判断；证据不足时改成自查问题。
- “修改次数”“报价表”“交付表”“变更表”不能默认出现。只有读者职业/诉求明确涉及服务交付、自由职业、项目制合作、咨询、设计、内容、技术或客户制业务，才使用这些具体商业词；否则改成“变更规则”“权责说明”“合作口径”等更通用表达。
- 方位、空间、城市和开运建议只在用户明确关心时出现；没有现场勘测和执行反馈时，只写低风险建议，不做风水强断。简洁版最多写 1 个小节，顺序固定为“更适合试什么 / 为什么 / 怎么低成本试 / 观察什么”，不要写成摆件清单。
- 每个主要章节至少有 1 句可加粗的直白结论。加粗句优先写“你真正该记住什么 / 这件事在现实里怎么发生 / 现在最该调整什么”，不加粗术语清单、证据堆叠或内部边界句。
- 加粗不要机械地放在每章第一句；可以放在白话场景、段落中后段或章节收束。正文重点只做加粗加深，不加底纹、不换字体、不额外跳色。
- 不做医疗、法律、投资或重大婚恋决定；结尾保留边界句。

## 推荐结构

### 先说最像你的地方

用 3-5 段直接进入读者反应模式。可以适度使用“你是不是……”式校准问题，但不要为了形式堆问句。

### 专业锚点

用 1 段写清核心事实，例如：

```text
几个专业锚点可以很简洁地抓住你：四柱是……，当前大运是……，紫微重点是……，西占稳定落座是……。所以这版不是泛泛描述，而是在说：……
```

只写稳定项和对正文有用的锚点；时间敏感项写成“需要校准”，不要写死。

### 底层矛盾

写这个人反复出现的内在拉扯：关系、安全感、规则、表达、钱、自由、责任、精力等。要写成现实语言，不写成术语解释。

### 关系与感情

写亲密关系、朋友、合作中的表现、误读、风险和更好的做法。不要把性别当作模板；按命盘看是偏表达难、承压、慢热、安全感、控制感、自由感还是责任感。

### 事业和钱

写适合的工作方式、价值呈现、定价/预算/合作边界。商业价值要先落到作品、案例、流程、权责、回款、复盘；只有读者确实处在项目制或客户制服务场景时，才写报价、交付范围和修改次数。

### 未来三到五年

按大运和流年写趋势，不写绝对事件。每年最多 1-2 段，重点是：主题、容易触发的议题、建议动作。

### 自查问题

列 5-8 个可验证的问题，帮助读者判断是否贴合。问题应来自命盘结构，而不是泛泛人格测试。

### 收束建议

用 2-4 段收束：读者最该拿走的边界、行动和提醒。最后写明不替代医疗、法律、投资和重大关系决定。

## 打包命名

源稿：

```text
<RUN_DIR>/drafts/<case>-concise-report.md
```

读者交付：

```text
<RUN_DIR>/delivery/<昵称>命盘简洁版.md
<RUN_DIR>/delivery/<昵称>命盘简洁版.docx
<RUN_DIR>/delivery/<昵称>命盘简洁版.pdf
<RUN_DIR>/delivery/<昵称>命盘简洁版-手机阅读.html
```

推荐命令：

```powershell
python scripts/package_reader_delivery.py <RUN_DIR>\drafts\<case>-concise-report.md --output-dir <RUN_DIR>\delivery --basename <昵称>命盘简洁版 --no-subtitle --avoid-locked --json --manifest <RUN_DIR>\case_manifest.json --artifact-prefix concise
python scripts/package_mobile_html.py <RUN_DIR>\delivery\<昵称>命盘简洁版.md --output <RUN_DIR>\delivery\<昵称>命盘简洁版-手机阅读.html --manifest <RUN_DIR>\case_manifest.json --artifact-key concise_mobile_html
```

## 简洁版验收

- 正文 5500-9000 字符为佳。
- 第二人称充足，读起来像个人报告，不像排盘说明。
- 至少包含 1 段专业锚点。
- 至少覆盖关系/感情、事业、钱、未来三到五年。
- 至少有 6-10 处自然分布的加粗结论句，重点是白话判断、行动建议和章节收束，不是术语和证据清单。
- 不出现“小红书”“情绪价值”“真实感增强”等内部讨论词。
- 不出现脚本路径、JSON 路径、命令、截图路径、内部验收说明。
- 草稿阶段先用 `scripts/validate_longform_report.py --profile reader-concise` 检查长度、第二人称、核心专题、自然加粗和内部讨论词；再用 `scripts/audit_longform_consistency.py` 非 strict 口径检查事实硬失败，strict 警告可作为丰富版模板差异保留。
- 手机阅读 HTML 默认生成，必须是单文件、移动端 viewport、暖黄纸感、仿宋/宋体优先、单列滚动、不加载外部资源；UI 遵循电子书长文标准：暖纸、深字、少色、无框、无卡片，默认不显示“简洁版报告”一类副标题，H1/H2 冒号后可拆两行但同字号同色同字重，H3 不额外跳色，正文加粗只用 700 字重。
- 单盘手机阅读器同样执行这个标准：H1/H2 都优先在冒号后自然分行，不在逗号处强行造断句，不用 `overflow-wrap: anywhere` 硬切中文词组；正文如果源稿漏标重点，可以每节最多补 1 个直白结论句为 700 字重加粗，自动补重点优先选第二段以后或章节收束句，不抢每章第一段，但不加底纹、不加荧光笔、不换色。
- 手机阅读器里的加粗句必须短、直白、可扫读，分散到多个章节；不要把术语段、证据段、免责声明或整段正文加粗。
- 手机阅读 HTML 必须回写 manifest：单盘简洁版使用 `concise_mobile_html`，合盘 rich 手机版使用 `relationship_mobile_html`，合盘简洁手机版使用 `relationship_concise_mobile_html`。
