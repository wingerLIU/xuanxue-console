# Xuanxue Console 项目偏好

- 面向读者的命理解读默认采用“证据约束下的锋利判断”：先给明确结论、倾向排序和不适合项，再讲证据、边界和校准问题。
- 客观不等于中庸。如果结构化 facts、知识库规则和现实校准支持某个方向明显更强，直接写“更像 A，不是 B”；不要为了稳妥把每个问题写成两边都可以。
- 锋利不是恐吓、绝对化或宿命化。医疗、法律、投资、重大婚恋决定仍必须降级为边界、现实建议和可校准问题。
- 合盘/关系报告必须先生成 run-local relationship facts，再写读者正文；正文只允许使用已知现实边界作为落点，不能把用户随口提供的关系细节扩展成命理事实。
- 合盘/关系报告优先用 `scripts/create_relationship_workspace.py` 从双方已有单盘 `case_manifest.json` 新建外部 relationship run；不要直接在旧合盘稿上改，也不要把 relationship facts 写进项目仓库。
- 合盘/关系报告默认第三人称书写，先写吸引、互补和可经营价值，再写张力、误读和校准问题；避免把张力写成劝退结论。
- 合盘丰富版默认使用 `templates/relationship-rich-template.md` 的 22 章结构；必须覆盖格局核对、神煞/辅助象意、亲密与私密边界、事业与合作交集、家庭与生活承载、财富资源边界、健康精力、关系形态变化和 MBTI 行为链条，但不得为了长度拆出重复章节。
- 情侣合盘可以写私密生活，但只能写成年关系中的身体吸引、亲密节奏、安全感、尊重和沟通边界；不得写露骨细节，不得推断具体频率或具体行为。
- 合盘里的神煞、纳音、旬空、十二长生只作辅助象意，不能覆盖日主、月令、十神、合冲刑害、大运流年和现实校准。
- 合盘/关系报告交付前运行 `scripts/validate_relationship_report.py`；校验不过时不得打包为读者交付。
- 单盘和合盘的手机阅读 HTML 使用同一 UI 标准：暖纸、深字、少色、无框、无卡片；H2 冒号后可拆两行但两行同字号同色同字重；正文重点只做 700 字重加粗，不加底纹、不换字体、不额外跳色。
- 回答追加问题时，先运行并读取 `scripts/create_followup_context.py` 生成的 run-local context；旧报告正文只能作为表达上下文，不能单独作为新判断来源。
- 追加问题和读者反馈默认归档到 `<RUN_DIR>/calibration/dialogue/`；只把去隐私、人工确认过的复盘候选晋升到知识库。
