# Team Career Synastry Knowledge Card

source_id: `SRC-PROJECT-TEAM-CAREER-SYNTHESIS`

多人事业合盘用于判断团队、合伙、创业、项目合作和城市选择。它不是双人恋爱合盘的放大版，也不是把每个人的优缺点相加。

## 证据层级

1. 单盘 facts：每个成员先有独立 `case_manifest.json`、结构化 JSON 和事实复查档案。
2. 两两 relationship facts：关键成员之间先有 pair run，至少覆盖合作、钱账、精力、家庭/生活承载和互动边界。
3. 团队现实资料：角色分工、决策权、否决权、钱账方式、交付责任、客户来源、城市候选和试点周期。
4. 团队级 summary：团队判断必须落到 run-local `team-source-summary.json`，不能只从旧报告或聊天印象里续写。
5. 命理辅助：八字、紫微、西占和 MBTI 只能辅助解释团队能量、沟通方式、角色错位和时间节奏，不能替代商业验证。

## 判断顺序

先写商业结构，再写命理辅助：

- 谁适合拍板，谁适合建议，谁适合交付，谁适合对外。
- 哪些冲突是职责不清，哪些是节奏不合，哪些可能是资源和钱账边界问题。
- 城市判断先看客户、供应链、行业密度、现金流、生活承载和试点成本，再用命理象意做辅助提示。
- 流年流月用于安排试点、谈判、上线、复盘和暂停，不用于承诺具体收益。
- 流日只适合轻量提示和日程选择，不进入主报告强判断。

## 禁止写法

- 不把“某人很强”“某人能破局”写成主观背书。
- 不把用户反馈直接改写成命理事实。
- 不把多人事业合盘写成恋爱合盘、家族关系评判或人格诊断。
- 不写“必赚”“必散伙”“一定适合某城市”等确定性结论。
- 不用单个成员的强项覆盖团队的权责、钱账和交付结构。

## 复盘采集点

团队合盘最需要真实反馈来校准。每次交付后优先收集：

- 哪个角色判断被现实验证，哪个角色判断误读。
- 哪个冲突点其实来自权责、钱账或沟通，而不是命盘张力。
- 城市判断里哪些商业变量比命理象意更关键。
- 流年流月建议是否帮助安排试点和复盘，还是显得空泛。
- 新成员加入后，是补短板、放大冲突，还是改变团队权力结构。

候选复盘只能进入外部 run 的 `retrospectives`。人工确认、去隐私、带反例和限制后，才能晋升到 `knowledge/case-retrospectives/`。
